# Notifications

## Overview

Notifications expose user-facing status changes and alerts. They are user-scoped and support read/unread tracking. Notifications are created by backend processes (scheduled jobs, system events) and can also be created via API for system integrations.

## Notification Types

| Type | Source | Description |
|------|--------|-------------|
| `GOAL_OFF_TRACK` | Goal progress scheduler | A tracked goal is falling behind its target |
| Various | System events | Generated by backend processes and scheduled jobs |

## Fields

| Field | Notes |
|-------|-------|
| `title` | Short summary of the notification |
| `message` | Detailed description with context |
| `type` | Notification category identifier |
| `read` | Boolean, defaults to `false` |
| `metadata` | Optional JSON payload with additional context (e.g., goal ID, budget ID) |

## Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/v1/notifications` | Create a notification (system use) |
| GET | `/api/v1/notifications` | List notifications (shared paginated response, sorted by recency) |
| GET | `/api/v1/notifications/{id}` | Get notification details |
| PUT | `/api/v1/notifications/{id}` | Update a notification |
| DELETE | `/api/v1/notifications/{id}` | Delete a notification |
| GET | `/api/v1/notifications/unread-count` | Get count of unread notifications |
| PATCH | `/api/v1/notifications/mark-all-read` | Mark all as read for the current user |

## List Response Shape

`GET /api/v1/notifications` returns the shared pagination contract from [API Reference](../api-reference.md). Notification records are returned in `items`.

## Unread Count

```bash
curl -X GET "http://localhost:8080/api/v1/notifications/unread-count" \
  -H "Authorization: Bearer <accessToken>"
```

Response:

```json
{
  "unreadCount": 3
}
```

## Mark All Read

```bash
curl -X PATCH "http://localhost:8080/api/v1/notifications/mark-all-read" \
  -H "Authorization: Bearer <accessToken>"
```

Sets all unread notifications for the current user to `read = true`.

## Push Delivery (FCM)

Every notification created via `NotificationService.create()` also triggers a push attempt to all of that user's registered devices (see `POST/DELETE /api/v1/users/me/device-tokens`). Delivery is via FCM HTTP v1, gated by `push.fcm.enabled` (default `false`) — see [Environment Reference](../env-reference.md#push-notifications-fcm).

The FCM `data` payload sent to the device is:

| Key | Value |
|-----|-------|
| `type` | The notification's `type` (e.g. `GOAL_OFF_TRACK`) |
| `<metadata key>` | Each top-level scalar field from the notification's `metadata`, stringified (e.g. `goalId`) |

Mobile clients route a tapped push using the `type` key the same way they'd resolve a `NotificationResponse.type` from the list endpoint.

## Error Codes

| Code | HTTP | When |
|------|------|------|
| `NOTIFICATION_NOT_FOUND` | 404 | Notification not found or not owned by the caller |

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Read/unread tracking with bulk mark-all-read | Common UX pattern; efficient for mobile clients |
| Metadata as JSON payload | Flexible — different notification types carry different context |
| User-scoped isolation | Each user sees only their own notifications |
